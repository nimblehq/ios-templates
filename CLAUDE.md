## Overview

This is Nimble's iOS template repository — a SwiftUI-based project scaffold powered by **Tuist 4** and **Swift Package Manager**. The template is generated into new projects using a Swift command-line tool (`iOSTemplateMaker`). The codebase uses placeholder names like `{PROJECT_NAME}` in many places that get substituted during generation.

## Generating a New Project

```bash
swift run --package-path Scripts/Swift/iOSTemplateMaker iOSTemplateMaker make \
  --bundle-id-production co.nimblehq.project \
  --bundle-id-staging co.nimblehq.project.staging \
  --project-name ProjectName \
  --interface SwiftUI
```

## Common Commands

**Generate Xcode project (required before opening in Xcode):**
```bash
tuist generate
```

**Install Ruby dependencies:**
```bash
bundle install
```

**Run tests:**
```bash
bundle exec fastlane buildAndTestLane
```

**Lint:**
```bash
swiftlint
```

**Build lanes:**
```bash
bundle exec fastlane buildAdHocStagingLane
bundle exec fastlane buildAdHocProductionLane
bundle exec fastlane buildAndUploadToFirebaseLane    # staging
bundle exec fastlane buildAndUploadToTestFlightLane
bundle exec fastlane buildAndUploadToAppStoreLane
```

Tool versions are managed via `.mise.toml` (Tuist 4.110.3, xcbeautify).

## Architecture

The project follows a clean, modular architecture with three layers:

### Modules (SPM-based frameworks)
- **`Modules/Domain/`** — Pure Swift business logic: use cases, entities, repository interfaces. No framework dependencies.
- **`Modules/Data/`** — Implements domain interfaces: network repositories (Alamofire), local storage (KeychainAccess).

### Main App Target
- **`Sources/Application/`** — App entry point, AppDelegate
- **`Sources/Presentation/`** — SwiftUI views, view models, coordinators, styles, shared components
- **`Sources/Constants/`** — API endpoints, feature flags
- **`Sources/Supports/`** — Extensions, helpers, builders

### Test Targets
- **`{PROJECT_NAME}Tests/`** — Unit tests using Quick + Nimble; mocks auto-generated by Sourcery; network stubbing via OHHTTPStubs
- **`{PROJECT_NAME}KIFUITests/`** — KIF-based UI tests organized into Screens, Flows, Specs

## Project Configuration

Tuist manifests drive all project generation:
- **`Project.swift`** — Root Tuist project manifest (targets, schemes, SPM packages)
- **`Package.swift`** — SPM dependency declarations
- **`Tuist/ProjectDescriptionHelpers/`** — Reusable Tuist configuration: `Module.swift`, `BuildConfiguration.swift`, `Target+Initializing.swift`, `Scheme+Initializing.swift`

### Build Configurations
Four configurations exist: `Debug Staging`, `Debug Production`, `Release Staging`, `Release Production`. XCConfig files in `Configurations/` define per-environment settings. Compilation flags: `DEBUG`, `STAGING`, `PRODUCTION`.

## Fastlane (Swift DSL)

All fastlane code uses the **Swift DSL** (`.swift` files, not Ruby). Key files:
- **`fastlane/Fastfile.swift`** — Lane definitions
- **`fastlane/Constants/Constant.swift`** — Project name, bundle IDs, device list, Firebase app IDs
- **`fastlane/Constants/Secret.swift`** — Environment variable mappings (never hardcoded secrets)
- **`fastlane/Helpers/`** — Manager classes: `Build.swift`, `Distribution.swift`, `Match.swift`, `Test.swift`, `Version.swift`

## CI/CD

Three platforms are supported: **GitHub Actions** (`.github/workflows/`), **Bitrise** (`bitrise.yml`), **CodeMagic** (`codemagic.yaml`). Code signing uses `match`. Distribution targets: Firebase Distribution (staging/QA), TestFlight (beta), App Store (production).
